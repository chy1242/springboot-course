# 基础
## 概念
**<font style="color:rgb(64, 64, 64);">图表代码化（Diagram as Code）</font>**<font style="color:rgb(64, 64, 64);"> 是一种通过编写代码（而非手动拖拽图形）来定义和生成图表的技术。它将图表的元素、布局和关系用结构化的文本描述，再通过工具自动渲染成可视化图形。</font>

## <font style="color:rgb(64, 64, 64);">典型工具</font>
| **<font style="color:rgb(64, 64, 64);">工具</font>**     | **<font style="color:rgb(64, 64, 64);">适用场景</font>**     | **<font style="color:rgb(64, 64, 64);">示例语言</font>**     |
| -------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **<font style="color:rgb(64, 64, 64);">PlantUML</font>** | <font style="color:rgb(64, 64, 64);">UML图、架构图</font>    | `**<font style="color:rgb(64, 64, 64);background-color:rgb(236, 236, 236);">@startuml\nA -> B: 交互\n@enduml</font>**` |
| **<font style="color:rgb(64, 64, 64);">Mermaid</font>**  | <font style="color:rgb(64, 64, 64);">流程图、时序图、甘特图</font> | `**<font style="color:rgb(64, 64, 64);background-color:rgb(236, 236, 236);">mermaid\ngraph LR\nA-->B\n</font>**` |
| **<font style="color:rgb(64, 64, 64);">Graphviz</font>** | <font style="color:rgb(64, 64, 64);">关系图、树状图</font>   | `**<font style="color:rgb(64, 64, 64);background-color:rgb(236, 236, 236);">digraph { A -> B }</font>**` |
| **<font style="color:rgb(64, 64, 64);">Diagrams</font>** | <font style="color:rgb(64, 64, 64);">云架构图</font>         | <font style="color:rgb(64, 64, 64);">Python 代码定义 AWS/GCP 资源</font> |


## <font style="color:rgb(64, 64, 64);">与传统绘图方式的对比</font>
| **<font style="color:rgb(64, 64, 64);">维度</font>**     | **<font style="color:rgb(64, 64, 64);">图表代码化</font>**   | **<font style="color:rgb(64, 64, 64);">传统方式</font>**     |
| -------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **<font style="color:rgb(64, 64, 64);">编辑方式</font>** | <font style="color:rgb(64, 64, 64);">写代码</font>           | <font style="color:rgb(64, 64, 64);">拖拽图形+手动调整</font> |
| **<font style="color:rgb(64, 64, 64);">协作效率</font>** | <font style="color:rgb(64, 64, 64);">适合团队协作（代码合并）</font> | <font style="color:rgb(64, 64, 64);">需文件传输或锁定编辑</font> |
| **<font style="color:rgb(64, 64, 64);">维护成本</font>** | <font style="color:rgb(64, 64, 64);">修改代码即可更新图表</font> | <font style="color:rgb(64, 64, 64);">需重新调整图形布局</font> |
| **<font style="color:rgb(64, 64, 64);">适用场景</font>** | <font style="color:rgb(64, 64, 64);">技术文档、开发设计</font> | <font style="color:rgb(64, 64, 64);">非技术用户、演示文稿</font> |


# PlantUML 和 <font style="color:rgb(64, 64, 64);">Mermaid</font>
## <font style="color:rgb(64, 64, 64);">是什么</font>
[**PlantUML**](https://plantuml.com/zh/)<font style="color:rgb(64, 64, 64);"> 是一个开源的</font>**<font style="color:rgb(64, 64, 64);">文本建模工具</font>**<font style="color:rgb(64, 64, 64);">，通过简单的代码语法生成多种类型的图表。它基于 </font>**<font style="color:rgb(64, 64, 64);">DSL（领域特定语言）</font>**<font style="color:rgb(64, 64, 64);">，将纯文本描述转换为可视化图形。</font>

[**Mermaid**](https://www.mermaidchart.com/)<font style="color:rgb(64, 64, 64);"> 是一个</font>**<font style="color:rgb(64, 64, 64);">基于 JavaScript 的图表生成工具</font>**<font style="color:rgb(64, 64, 64);">，允许用户通过简单的文本语法（类似 Markdown）创建多种类型的图表，并直接在网页或文档中渲染为可视化图形。</font>

## <font style="color:rgb(64, 64, 64);">两者对比</font>
| **指标**   | **PlantUML**                               | **Mermaid**                                |
| ---------- | ------------------------------------------ | ------------------------------------------ |
| 语法复杂度 | 略复杂、功能全面，细节控制强               | 语法极简、易入门，适合快速文档/教学        |
| 支持图类型 | UML（用例、时序、类、ER、组件图）全覆盖    | 流程图、类图、时序、ER、思维导图等         |
| 渲染环境   | VSCode/PlantUML插件、在线编辑器、Java 环境 | Markdown编辑器、Typora/Obsidian/Notion原生 |
| 导出与集成 | PNG、SVG、PDF、可与 CI/CD 集成、排版严谨   | Markdown嵌入、GitHub/Gitee/Typora 原生显示 |
| 图表样式   | 工程化、标准出版级别、模板丰富             | 简洁美观，适合知识库、讲义、作业           |
| 推荐场景   | 正式论文、工程文档、复杂系统建模           | 教学、博客、轻量级报告、线上文档           |


**建议：**

+ 正式论文、工程报告、复杂架构文档，推荐 PlantUML；
+ 日常学习、博客、轻量型报告、教学课件优先 Mermaid；
+ 实训/作业可两者并用，加深对“代码即图表”理念的理解。

## 用例图（Use Case Diagram）
+ **目的**：分析平台各类用户的功能需求
+ **要求**：用例图需完整展现运营管理员、租户管理员、用户及其典型操作，注重主次分明、功能聚焦。

### 基础要素
| 要素                        | 说明                                                         | PlantUML 符号示例                                            |
| --------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 参与者（Actor）             | 与系统交互的用户、外部系统、设备等，位于系统边界之外         | `actor 用户`<br/>                                            |
| 用例（Use Case）            | 系统为参与者提供的功能或服务，用椭圆表示，放在系统边界内部   | `(添加设备)`<br/>                                            |
| 系统边界（System Boundary） | 用矩形框表示系统范围，所有用例放在框内，参与者在外           | `rectangle "物联网平台" { ... }`<br/>                        |
| 关系（Relationship）        | 参与者与用例、用例与用例之间的连线，包括关联、include、extend、泛化等 | `用户 --> (添加设备)`<br/>`(用例A) --> (用例B) : <<include>>`<br/>`(用例A) --> (用例C) : <<extend>>` |
| 注释（Note）                | 用于补充说明用例或关系，可用note标注                         | `note right of (添加设备)\n包含扫码、手动录入等`             |

## ER 图（Entity-Relationship Diagram）
+ **目的**：设计清晰的数据表结构
+ **要求**：ER 图需体现租户、用户、设备、设备类型、告警等主表及其主外键关系，字段名和关系一目了然。

### 基础要素
| 要素                       | 说明                                                         | PlantUML 符号示例                             |
| -------------------------- | ------------------------------------------------------------ | --------------------------------------------- |
| 实体（Entity）             | 需要建模的对象，通常为表，对应一个集合，如用户、设备等。     | `entity 用户 { ... }`<br/>方框                |
| 属性（Attribute）          | 实体的具体特征或字段，常见主键、外键、一般属性，主键常用*或+标记。 | `*用户ID : bigint`<br/>`用户名 : varchar`     |
| 主键（Primary Key）        | 唯一标识实体的属性，常用*或+标注，ER图中常加下划线或特殊标记。 | `*设备ID : bigint`<br/> 或 `+设备ID : bigint` |
| 外键（Foreign Key）        | 关联其他实体的属性，用于建立表间关系，常用FK后缀或特殊标记。 | `租户ID : bigint`                             |
| 实体间关系（Relationship） | 两个或多个实体间的逻辑联系，如1:1、1:N、N:M。                | `租户                                         |
| 关系名称                   | 标明实体间的联系含义，ER图线上用中文/英文标注，如“管理”“拥有”。 | `: 管理`                                      |
| 关系基数（Cardinality）    | 描述实体参与关系的数量：一对一（1:1）、一对多（1:N）、多对多 |                                               |

## 时序图（Sequence Diagram）
+ **目的**：描述典型业务操作的流程与各系统/参与者间的交互
+ **要求**：以“设备上报告警”为例，时序图清楚表达每一方操作顺序及数据流，突出 IoT 特性。

### 基础要素
| 要素                        | 说明                                                         | PlantUML 符号示例                      |
| --------------------------- | ------------------------------------------------------------ | -------------------------------------- |
| 参与者（Actor/Participant） | 参与交互的对象，包括人、系统、组件等。用actor或participant标记。 | `actor 用户`<br/>`participant 设备`    |
| 生命线（Lifeline）          | 显示参与者在交互过程中的存在，通常为垂直虚线，由actor/participant自动生成。 |                                        |
| 消息（Message）             | 对象之间传递的信息、调用、响应等，用箭头表示，说明消息内容。 | `A -> B: 请求信息`<br/>`B --> A: 响应` |
| 同步/异步消息               | 同步用`->`，异步用`->>`，响应用`-->`等。                     | `A ->> B: 异步消息`                    |
| 激活（Activation）          | 表示对象正在执行操作，通常为生命线上的矩形条，PlantUML自动管理。 |                                        |
| 返回消息                    | 表示操作的返回，用虚线箭头。                                 | `B --> A: 返回结果`                    |
| 注释（Note）                | 说明或补充业务流程、特殊步骤，贴在某个生命线或消息旁边。     | `note right of B ... end note`         |
| 分区（alt/opt/loop）        | 控制流程的分支、循环等，用alt/opt/loop结构表示。             | `alt ... end`<br/>`loop ... end`       |

## 流程图
+ **目的：**规范描述业务处理的每一个步骤和决策点
+ **要求：**以“设备告警处理”为例，流程图结构需清晰、判断环节分明，要求符合 ISO 5807 标准。

### 基础要素
| 要素                                    | 说明                                              | PlantUML 标准符号示例                                   |
| --------------------------------------- | ------------------------------------------------- | ------------------------------------------------------- |
| 起始/终止（Terminal/Start/End）         | 表示流程的开始或结束，标准用椭圆（圆角矩形/椭圆） | `start`<br/>、`end`<br/>，或`(开始)`<br/>、`(结束)`     |
| 处理（Process）                         | 一般操作或处理步骤，标准用长方形                  | `:处理内容;`<br/>、矩形                                 |
| 判定/判断（Decision）                   | 二分/多分支条件判断，标准用菱形                   | `if (条件?) then (是)... else (否)... endif`<br/>，菱形 |
| 输入/输出（Input/Output）               | 数据输入或输出，标准用平行四边形                  | `input/output`<br/>、平行四边形                         |
| 流程线（Flow Line/Arrow）               | 步骤之间的流转方向，标准用带箭头的直线            | `-->`<br/>、`->`<br/>、箭头                             |
| 连接点（Connector）                     | 跳转或跨页连接，标准用圆圈内字母/数字             | `@label`<br/>，或圈内字母                               |
| 注释（Annotation）                      | 对某步骤补充说明，标准用带斜线的方框              | `note right of ... end note`                            |
| 子流程（Subprocess/Predefined Process） | 可复用或分解的子流程，标准用双线矩形              | 子流程符号                                              |

## 部署图（Deployment Diagram）
+ **目的：**展现系统主要模块、服务组件与交互关系
+ **要求：**体现平台前端、API 网关、后端服务（设备、用户）、数据库、MQTT 等模块的调用与依赖关系，突出分层和集成点。

### 基础要素
| 要素                           | 说明                                                 | PlantUML/符号示例                     |
| ------------------------------ | ---------------------------------------------------- | ------------------------------------- |
| 节点（Node）                   | 运行软件的物理或虚拟设备，如服务器、容器、手机等。   | `node "应用服务器"`<br/>`node "网关"` |
| 设备（Device）                 | 节点的物理载体，常与node等价，强调硬件本体。         | `node "数据库服务器"`                 |
| 组件（Component）              | 部署到节点上的可执行单元，如JAR包、服务实例、App等。 | `[设备服务]`<br/>`[Web前端]`          |
| 关系（Relationship）           | 节点、组件之间的通信/部署/依赖关系，用箭头或线标注。 | `node1 --> node2 : 网络连接`          |
| 通信链路（Communication Path） | 描述节点间网络或通信方式。                           | `node "A" ..> node "B" : HTTP`        |
| 数据存储（Artifact/Database）  | 以数据库或文件形式部署的持久化单元。                 | `database "主数据库"`                 |
| 部署（Deployment）             | 组件被分配到节点的映射关系，组件符号放在node块内。   | `node { [服务A] }`                    |
| 注释（Note）                   | 补充说明节点、组件、通信等业务或技术约束。           | `note right of node ... end note`     |


### 实践
:::info
请为多租户物联网平台画标准的 UML 部署图，要求包含以下要素：

+ 节点：Web 服务器、API 服务器、Nacos 服务器、 MQTT 网关服务器、数据库服务器、Redis 缓存服务器

结构要层级分明，为了保证所有线条是直线，可以只保留核心依赖线，或者分两张图分层展示。
